## 先问三个问题

在决定用什么之前，问自己三个问题：

1. **任务数量**：是 1-3 个独立任务，还是几十个任务？
2. **可提前规划**：执行流程可以提前写成脚本，还是需要 LLM 动态判断？
3. **需要上下文**：子任务需要父会话的对话历史，还是独立可执行？

## 决策流程

| 任务数量 | 可提前写脚本 | 需要上下文 | 推荐方案 |
|---------|------------|-----------|---------|
| 1-3 个 | 否 | 是 | Fork 子代理 |
| 1-3 个 | 否 | 否 | 同步子代理（或后台） |
| 1-3 个 | 是 | — | 小规模工作流 |
| 几十个 | 是 | — | ultracode 工作流 |
| 几十个 | 否 | — | 不适合自动化 |

## 功能对比矩阵

| 特性 | 同步子代理 | Fork 子代理 | 后台子代理 | ultracode |
|------|----------|-----------|----------|----------|
| 上下文继承 | 无 | 完整 | 无 | 无 |
| 主会话阻塞 | 是 | 是 | 否 | 否 |
| 最大并行数 | 3 | 1 | 3 | 无限制 |
| 可重跑 | 否 | 否 | 否 | 是 |
| 需要写脚本 | 否 | 否 | 否 | 否（Peri 生成） |
| 适合规模 | 少量任务 | 单次协作 | 少量独立任务 | 大规模批量 |

## 场景速查

| 你想做什么 | 用什么 | 怎么用 |
|-----------|--------|--------|
| 搜索代码库中某个模式 | explorer 子代理 | 对 Peri 描述搜索目标 |
| 改代码后立即审查 | code-reviewer 子代理 | 改完代码后让 Peri 审查 |
| 审查 + 修复同步进行 | 后台模式 coder + code-reviewer | 对 Peri 说"同时改和审查" |
| 先研究结构再写代码 | Fork 模式 | 先让 explorer 探索，然后 Fork coder |
| 扫描 50 个文件找 dead code | ultracode | `/ultracode 扫描所有文件找 dead code` |
| 多角度审查一批变更 | ultracode | `/ultracode 从安全/性能/风格审查` |
| 批量迁移文件格式 | ultracode | `/ultracode 把 class 组件改成函数组件` |

## 成本考量

| 方案 | Token 消耗 | 执行时间 |
|------|----------|---------|
| 同步子代理 (haiku) | 低 | 快 |
| 同步子代理 (sonnet) | 中 | 快 |
| Fork 子代理 | 中高（复制上下文） | 中 |
| 后台子代理 | 中 | 异步，不感知 |
| ultracode | 高（多 agent 总和） | 长（但并行） |

<Aside type="tip">
  日常开发和搜索优先用子代理。批量操作和复杂流程用 ultracode。不确定时先对 Peri 描述任务——Peri 会自己判断需要子代理还是工作流。
</Aside>
