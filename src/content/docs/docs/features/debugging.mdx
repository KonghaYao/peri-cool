## 诊断优先于修复

这是 peri 项目调试中反复验证的教训：**根因未被日志/代码证据定位前，禁止大改代码。**

```text
# ❌ 错误做法：直接动手改
"compact 重复了，我猜是 XXX 的问题，改一下"

# ✅ 正确做法：先加诊断日志
加一些 tracing 日志在 compact 入口，
看看是哪里触发的重复 compact，把调用栈打出来
```

添加诊断日志优先于修改业务逻辑。日志揭示的是事实，猜测揭示的是偏见——用事实迭代，不用偏见过日子。

## 3 次尝试上限

同一 bug 修复超过 3 次仍失败 → **禁止继续猜测式修复。** 这不是消极，是止损。

```text
# 第 3 次失败后
已经试了 3 个方向都不对。暂停修改代码。

请用 RUST_LOG=debug cargo run 2>&1 | grep compact
把运行时的完整日志给我，我基于数据重新做判断。
```

连续试错不但浪费 token，更会在代码里留下僵尸逻辑——已尝试的修复路径半吊子残留在各处。

## 用 /doctor 自检

出问题先判断是环境问题还是代码问题：

```text
/doctor
```

Peri 会检查：依赖版本一致性、配置文件完整性、构建缓存状态。不少"bug"其实是环境不对——花一小时改代码不如花一分钟跑 /doctor。

## 最小化复现

上下文被之前几十轮对话污染后，在新会话里重来：

```text
/clear
// 在新会话里用最小用例复现 bug
cargo test compact_single_file -- --nocapture
```

不带任何历史假设，从空白状态重新排查。成功率远高于在旧会话里死磕。

## 日志先行

不要只看错误信息。让 Peri 深入读日志：

```text
搜 compact 相关的所有 tracing span，
列出完整调用链和每次 compact 的触发位置及入参
```

Peri 能并行 grep、交叉引用、追踪调用链——比人肉眼扫日志快一个数量级。

<Aside type="tip">
  复杂 bug 的真实调试中，诊断阶段平均消费 10-20 条消息，修改阶段往往不到 5 条。投入在诊断上，最终快得多。
</Aside>
