## 并发限制

Peri 默认最多 3 个并行子代理。这是硬限制，超过会排队。

对于免费版 API（如 DeepSeek、GLM 免费额度），建议限制为 2 个并行，避免触发速率限制。你可以在任务描述中指定："最多 2 个并行"。

## 模型选择策略

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 代码搜索 | haiku | 搜索不需要强推理，便宜够用 |
| 网络研究 | haiku | 搜索和摘要任务，轻量模型即可 |
| 代码编写 | sonnet | 需要较强的代码理解能力 |
| 代码审查 | sonnet | 需要识别细微的逻辑问题 |
| 方案设计 | 继承主会话 | 方案质量取决于所用模型 |
| 实现验证 | sonnet | 需要准确解读构建和测试输出 |

自定义子代理中指定 model 可以永久绑定模型策略，不依赖主 Agent 的选择。

## 任务描述规范

子代理的任务描述（prompt）直接影响执行质量。

**具体而非笼统。** 不是"审查这个项目"，而是"审查 src/services/ 下 3 个文件的错误处理逻辑"。

**包含输出期望。** 不是"分析代码"，而是"输出问题清单，每项包含：位置、严重程度、修复建议"。

**给出边界。** 不是"修改全部文件"，而是"只修改 src/api/ 下的文件，不动 src/db/"。

**任务最多 8 个文件。** 单次 Agent 调用最多处理 8 个文件，超过则拆分成多个批次。

## 文件操作约束

给 coder 子代理写 prompt 时，必须**使用绝对路径**。子代理不遵守 `cwd` 参数，相对路径会导致操作目标错误。

每个 coding prompt 应包含文件白名单和禁止清单：

```
待修改文件：
- src/services/auth.rs
- src/services/session.rs

DO NOT modify:
- src/db/
- Cargo.toml
```

<Aside type="caution">
  多个子代理并行修改相同文件会导致冲突。如果任务涉及共享文件，用串行执行或 Fork 模式。
</Aside>

## 批量替换

超过 10 个文件的模式替换（重命名、import 路径等），不要派子代理一个个 Edit。直接在 prompt 中明确指示用脚本批量处理：

```bash
find src/ -name "*.rs" -exec sed -i '' 's/old_module/new_module/g' {} +
```

让子代理逐个 Edit 上百个文件既慢又容易出错。

## 常见陷阱

### 不要在同一批文件中并行写

多个子代理并行修改相同的文件会导致修改冲突。共享文件的操作必须串行化。

### 不要创建过多的自定义子代理

为每个微小差异创建独立子代理会导致 Peri 难以选择。优先设计几个通用子代理，用任务描述区分行为。

### 注意后台任务的完成时序

后台模式下子代理完成后才推送通知。如果你依赖子代理结果做下一步操作，使用同步模式。

<Aside type="tip">
  后台任务结果通过 system-reminder 自动推送，不要写循环轮询 AgentResult——浪费 token 且结果会重复推送。
</Aside>

### 后台任务需要构建验证

后台 agent 完成后，应在 prompt 中要求它 `cargo build` 验证自己的改动。验证失败则自动回滚修改（`git checkout -- <modified files>`），避免把编译不过的代码留在仓库里。

### 避免 Fork 模式嵌套过深

Fork 模式的子代理不能再 Fork。如果需要多层上下文传递，考虑用 ultracode 工作流。
