## 审查不是一步，是三层

别只用一句话让 Peri "审查一下代码"。实际生产中的审查是**三层递进**的：

```text
# 第一层：主代理写完代码后自我审查（这是最弱的）
我刚改了 src/agent/prompt.rs 里的上下文注入逻辑，帮我自查一遍。

# 第二层：派 coder 子代理从空白上下文重新审查
派 coder 子代理审查 src/agent/prompt.rs 的改动，
它没见过之前的对话，不会有我对话历史里的假设。

# 第三层：派 verification 子代理做独立终验
派 verification 子代理做最终审查，
检查：1. 类型安全 2. 错误处理路径 3. 有没有忘了更新调用方。
```

核心思路：**每层用干净的上下文打断上一层的思维惯性。** 主代理容易"确认偏误"——它刚写出代码，潜意识里觉得没问题。子代理从空白启动，更容易发现遗漏。

## 对抗性审查 /ultracode（Peri 内置）

真实对话模式——我经常这样用：

```text
/ultracode 并行审查我刚改的代码：
1. explorer 搜索所有调用方，确认接口兼容
2. coder 审查核心逻辑——关注错误处理和边界条件
3. code-reviewer 用不同角度独立审查，贴上证据（代码片段、测试输出），不要断言
```

`/ultracode` 把三个子代理并行派出去。**三个代理从不同视角看同一份代码**——一个查调用链兼容性，一个看逻辑正确性，一个做规范审查。汇总的结果比单代理审查可信一个数量级。

## `/goal`（Peri 内置）设置审查标准

审查不是检查代码"好不好看"。审查前用 `/goal` 明确标准：

```text
/gcal 这次审查重点关注：
- 有没有可能导致 silent data corruption 的 bug
- 错误处理路径有没有遗漏（unwrap/expect 位置）
- 新增的 trait 实现会破坏已有调用方吗
```

把标准写清楚，代理不会偏离到格式化、命名这种低价值问题上——它只盯着你关心的事。

## 要求证据，不要断言

审查报告里最危险的写法是"看起来没问题"。这是废话。加上证据要求：

```text
审查 src/agent/session.rs 改动。
对每个发现，贴出：
- 相关代码片段（至少 5 行上下文）
- 为什么这里有问题（具体场景推演）
- 如果跑测试，哪个 case 会挂
```

Peri 贴出具体代码时，你自己扫一眼也能发现审查代理漏掉的东西。人和代理互相补位。

## 与分支相关审查

```text
# 对比 branch 和 main 的差异再审查
审查当前分支和 origin/main 的 diff，关注：
- API 兼容性变化
- 新引入的 unwrap/panic
- 有没有忘记迁移的调用方
```

<Aside type="tip">
  重要改动在 `/clear` 后的新会话中审查——上下文干净，不受之前几十轮调试的假设污染。重要审查严禁在调试的同会话进行。
</Aside>
